# Acceptance Matrix

This matrix defines verification-ready requirements from `spec.md` and `prd.json`, each with a stable requirement ID, measurable acceptance criteria, test linkage, and implementation status.

## Status legend

- **Not started**: no implementation evidence.
- **In progress**: partial implementation exists, but acceptance criteria are not fully met.
- **Done**: implementation appears complete, but criteria are not yet verified by automated/manual execution evidence.
- **Verified**: criteria have passing evidence from linked automated tests/benchmarks or completed manual validation.

## CI release-gate policy

A release is blocked unless all **must-have** criteria are in **Verified** status.

### Policy rules

1. Maintain this matrix as source-of-truth for requirement verification state.
2. CI MUST fail release jobs when any row with `Priority = Must-have` has `Status != Verified`.
3. CI MAY allow non-release branches to pass with `In progress` or `Done` for must-have criteria, but release/tag pipelines cannot.
4. Rows marked manual must include a rationale and a linked runbook or issue for automation follow-up.

### Suggested CI check implementation

- Add a parser script that reads `docs/acceptance-matrix.md` and validates:
  - all must-have rows have status `Verified` for release workflows;
  - status values are one of `Not started | In progress | Done | Verified`.
- Wire parser into release pipeline as a hard gate (non-zero exit on violation).

Example policy expression:

```text
if workflow == release:
  fail when exists(requirement where priority == must-have and status != Verified)
```
